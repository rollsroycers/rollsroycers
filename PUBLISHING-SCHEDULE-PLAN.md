# 🗓️ سيناريو جدولة نشر الـ700 مقال على 9 شهور (نشر عشوائي طبيعي)

> الهدف: إنشاء الـ700 مقال **دفعة واحدة**، ثم إظهارها للجمهور **تدريجيًا على ~9 شهور**، بأوقات **عشوائية خلال اليوم** (لا ميعاد ثابت) بحيث يبدو النشر بشريًّا وعضويًّا — لا دفعة آلية واحدة.

---

## 1) القيد الجوهري (لماذا التصميم بهذا الشكل)
الموقع **static (Next SSG + `fallback: false`)** على Cloudflare Workers: المقال **لا يظهر للجمهور إلا بعد `build + deploy`**. لا يوجد ISR/on-demand. إذن «النشر المجدول» = **التحكّم في أي المقالات تدخل البناء ومتى يحدث الـ deploy** — وليس مجرد تواريخ مخزّنة.

الحل المعتمد = **بوابة وقت + إعادة بناء عند الاستحقاق**:
- كل ملف مقال يحمل `publishAt` (من `publish-schedule.json`).
- **البناء يُظهر فقط** المقالات التي `publishAt ≤ وقت البناء` (في `getStaticPaths` + الـ sitemap عبر [`blogFileStore.ts`](src/data/blogFileStore.ts) و[`generate-sitemap.mjs`](scripts/generate-sitemap.mjs)).
- **cron كل 15 دقيقة** يشغّل [`publish-due.mjs`](scripts/publish-due.mjs): لو استحقّ مقال (مرّ وقته) ⇒ `build + deploy + IndexNow`؛ وإلا **no-op فوري**. فيظهر كل مقال خلال ≤15 دقيقة من وقته العشوائي ⇒ يبدو طبيعيًا. (شدّها لـ `*/10` لدقّة أعلى.)
- النتيجة: ~**2–3 deploy فعلي/يوم** في أوقات عشوائية، والباقي لا-عمليات رخيصة.

---

## 2) الجدول — كيف يبدو طبيعيًا (خوارزمية [`gen-publish-schedule.mjs`](scripts/gen-publish-schedule.mjs))
نتيجة فعلية (start `2026-08-01`, 9 شهور, seed 7):

| المؤشّر | القيمة | لماذا طبيعي |
|---|---|---|
| أيام نشطة | **257 / 274** (17 يوم هادئ) | لا نشر كل يوم — إجازات/فجوات |
| متوسط/يوم | **2.72** (المدى 1–5) | تباين، لا رقم ثابت |
| توزيع الأيام | 1→37 · 2→79 · 3→65 · 4→70 · 5→6 يوم | منحنى متموّج لا grid |
| نهاية الأسبوع (جمعة/سبت) | أخفّ | إيقاع تحرير واقعي في الإمارات |
| أوقات اليوم | عشوائية موزونة (ذروة 09–12 + مساء) + دقائق/ثوانٍ عشوائية، لا `:00` | مثال: `06:56 · 09:37 · 10:35 · 11:12` |
| تباعد داخل اليوم | ≥ 35 دقيقة | لا نشر متلاصق |
| التوزيع الشهري | 86·74·77·69·83·80·78·76·77 | متموّج لا مسطّح |

**ترتيب موضوعي (Topical Authority):**
- **المحاور (Pillars) تُنشر أولًا** (تنتهي ~الشهر الرابع)، ثم الدواعم (Supporting) على المدى.
- **العناقيد متشابكة (round-robin)** فلا أسبوع يغرق في موضوع واحد.
- 🔒 **ثابتة حرجة (مؤكَّدة برمجيًا = 0 خرق):** كل Supporting يُنشر **بعد** محوره ⇒ الرابط الصاعد للـ Pillar لا يشير أبدًا لصفحة لم تُنشر بعد.

## 3) تفاعل الجدولة مع الروابط الداخلية (نقطة عميقة)
بما أن النشر تدريجي، **رابط داخلي لمقال لم يُنشر بعد = صفحة مبوّبة (404)**. لذلك قاعدة الـ3–5 روابط تُقيَّد هكذا للمقالات المجدولة:
- ✅ **اربط دائمًا بما هو حيّ مضمونًا:** المحور (يُنشر قبلك) + صفحات دائمة (`/pricing`, `/booking`, `/fleet/<model>`, `/services/<x>`).
- ⚠️ **تجنّب الربط لمقال شقيق مجدول لاحقًا.** إن لزم، اربط فقط بمقال `publishAt` أبكر منك.
- البوابة في `blogFileStore`/sitemap تُخفي غير المنشور تلقائيًا، فحتى لو وُجد رابط مبكر فالوجهة مخفية حتى وقتها.

---

## 4) لماذا هذا أفضل لـ SEO
- **نمو tدريجي للـ sitemap** (روابط جديدة كل بضعة أيام) = إشارة نشر عضوي قوية لجوجل، لا «إغراق 700 رابط مرة واحدة».
- **`date` كل مقال = وقت ظهوره الفعلي** (يضبطه `apply-schedule.mjs`) ⇒ نضارة (freshness) صادقة في schema.
- **إيقاع ثابت** (~2–3/يوم) يحبّه جوجل أكثر من دفعة واحدة ثم صمت.
- **المحاور أولًا** تبني سلطة موضوعية قبل أن تتدفّق الدواعم إليها.

## 5) Runbook — الخطوات بعد إنشاء الـ700 + انتهاء الصور
```bash
# 1. ولّد الجدول بتاريخ البداية الحقيقي (بعد الصور). seed ثابت = جدول قابل لإعادة الإنتاج.
node scripts/gen-publish-schedule.mjs --start 2026-08-15 --months 9 --seed 7
#    → راجع الأرقام؛ غيّر --seed لو عايز نمطًا مختلفًا.

# 2. اطبع الـ publishAt + date داخل كل ملف مقال.
node scripts/apply-schedule.mjs --apply

# 3. تحقّق احترافي لكل المقالات (تكافؤ + i18n + روابط ≤5 + صور).
node scripts/blog-verify.mjs           # يجب أن يمرّ

# 4. (مرة واحدة) أنشئ صفحة الإظهار الأولى: deploy عادي — يظهر فقط ما استحقّ حتى الآن.
npm run deploy

# 5. فعّل الجدولة: GitHub Actions (.github/workflows/scheduled-publish.yml) كل 15 دقيقة،
#    أو cron محلي: */15 * * * *  cd repo && node scripts/publish-due.mjs --run
#    أسرار مطلوبة: CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID, INDEXNOW_KEY
```
`publish-due.mjs` يحفظ `.publish-state.json` (high-water mark) فلا يكرّر نشرًا، ويبلّغ IndexNow بالروابط الجديدة (en/ar/ru).

## 6) بديل — إظهار باللحظة الدقيقة (Edge-gating)
لو أردت ظهورًا **بالثانية** بلا إعادة بناء: ابنِ الـ700 كلها مرة واحدة، وأضِف منطقًا في Cloudflare Worker يردّ **404** لأي مقال `publishAt > now` (+ sitemap ديناميكي يكشف الروابط بمرور الوقت). أدقّ توقيتًا، لكنه أعقد (Worker + sitemap ديناميكي) و«ظهور خلال 15 دقيقة» يبدو طبيعيًا أصلًا. **التوصية: ابدأ بالبناء المجدول (الأبسط والأسلم).**
